# AI駆動開発シナリオ: 一問一答でタスク管理表を完成させる

このMarkdownは、実際のリポジトリを作るためのファイルではありません。  
非エンジニアのユーザーが、AIエージェントと一問一答で進めながら、最終成果物である「タスク管理表」を完成させるための課題シナリオです。

レッスン全体の入口は `index.md` です。  
このシナリオは、`github-login-setup-guide.md` 側のGitHub接続設定を確認したあとに使います。

## このMarkdownの使い方

このMarkdownは、学習者が直接読む教材であると同時に、エージェントがレッスンを迷わず誘導するための進行台本です。  
エージェントは、会話を始める前に全体を読み、現在どの段階を進めているかを確認してから一問一答で進行します。

エージェントが最初に確認する順番:

```text
1. index.md
2. github-login-setup-guide.mdの完了状態
3. tools/check_lesson_structure.sh
4. tools/lesson 現在地
5. 目的
6. 運用時のドキュメント分割
7. デモモードと実作業モード
8. 機械的な強制力
9. エージェント進行ルール
10. プロンプト実行体験の進め方
11. プロジェクトディレクトリとリポジトリの配置
12. 最小ファイル構成
13. 要件・仕様・実装計画を先に作る理由
14. 1週間の進行計画
15. 最初の一問一答シナリオ
```

章ごとの役割:

| 章 | エージェントが確認すること |
| --- | --- |
| index.md | レッスン全体の入口、設定から体験開発への順序 |
| 目的 | レッスンの主目的がAI駆動開発体験であること |
| 運用時のドキュメント分割 | 学習者向け、プロンプト集、テンプレート、進行台本の分担 |
| デモモードと実作業モード | 実ファイル作成をするか、説明だけにするか |
| 機械的な強制力 | `tools/lesson` でスキップ禁止、完了済み復習可を制御する |
| エージェント進行ルール | 一問一答、承認、プロンプト提示、禁止事項 |
| プロンプト実行体験の進め方 | ユーザーに何をコピペ実行してもらうか |
| プロジェクトディレクトリとリポジトリの配置 | 今扱うファイルがどこにあるか |
| 最小ファイル構成 | どのMarkdownや実装ファイルを使うか |
| 要件・仕様・実装計画 | 設計図と作業日誌をどう同期するか |
| 1週間の進行計画 | 今日どこまで進めるか |
| 最初の一問一答シナリオ | 初回にどの順番で質問するか |

判断に迷ったときの優先順位:

```text
1. 開発者の承認なしに書き込まない
2. 一問一答を守る
3. ユーザーにプロンプトを一から考えさせない
4. 作業の節目でフォルダ構成を見せる
5. 1週間で終わる小さい範囲に戻す
6. 最新の設計図と作業日誌の整合性を保つ
```

エージェントの1ターンの基本形:

```text
1. 現在のDayまたはStepを確認する
2. 今扱うファイルの位置を必要に応じて示す
3. 今やることを短く説明する
4. 質問を1つだけ出す
5. 必要ならコピペ用プロンプトを提示する
6. ユーザーの回答または実行結果を待つ
7. 次に進む前に、承認が必要な操作か確認する
```

途中から再開するときに確認する順番:

```text
1. 全体のフォルダ構成
2. tools/check_lesson_structure.sh
3. tools/lesson 現在地
4. LEARNING_HANDOFF.md
5. LEARNING_TASK_TRACKER.md
6. HANDOFF.md
7. TASK_TRACKER.md
8. REQUIREMENTS.md
9. SPECIFICATION.md
10. IMPLEMENTATION_PLAN.md
11. 今日進めるDayまたはStep
```

まだファイルが存在しない初回は、`最初の一問一答シナリオ` から始めます。  
ファイルが存在する再開時は、まず `tools/lesson 現在地` を実行し、順番上の現在位置を特定します。  
そのうえで、学習用の `LEARNING_HANDOFF.md` と `LEARNING_TASK_TRACKER.md` を確認し、学習者の理解状態を確認します。  
その後、開発作業用の `HANDOFF.md` と `TASK_TRACKER.md` を確認し、次の質問を1つだけ出します。

この構成にする目的は、学習者が自然にAI駆動開発を体験できるようにしながら、エージェント側も「次に何を聞くか」「何を作るか」「どこへ記録するか」で迷わないようにすることです。

## 運用時のドキュメント分割

このMarkdownは、レッスン全体を管理するマスター台本です。  
学習者にすべてを読ませるのではなく、実際のレッスンでは必要に応じて短い補助ドキュメントへ分けて使います。

分割する理由:

```text
学習者は今日使う部分だけを読める
エージェントは進行台本として全体を参照できる
プロンプト、テンプレート、実行手順を探しやすくなる
シナリオ説明と実行用手順が混ざりにくくなる
```

推奨する補助ドキュメント:

| ファイル | 読む人 | 目的 |
| --- | --- | --- |
| index.md | 学習者とエージェント | レッスン全体の入口、設定から体験開発への順番を示す |
| github-login-setup-guide.md | 学習者とエージェント | GitHub接続設定を一問一答で確認する |
| guides/LESSON_GUIDE.md | 学習者 | 今日やること、操作手順、確認ポイントだけを短く見る |
| prompts/PROMPTS.md | 学習者とエージェント | コピペ用プロンプトをDay別に探す |
| templates/TEMPLATES.md | エージェント | REQUIREMENTS.md、SPECIFICATION.md、IMPLEMENTATION_PLAN.mdなどの雛形を使う |
| playbooks/AGENT_PLAYBOOK.md | エージェント | 進行順、分岐、再開、失敗復旧の判断を確認する |
| lesson/LESSON_CONFIG.tsv | エージェント | リポジトリ名、配置、主要ファイル名の設定を確認する |
| lesson/LESSON_FLOW.tsv | エージェント | レッスン項目の正しい順番を機械的に確認する |
| learning/LESSON_STATE.tsv | エージェント | 現在のレッスン位置、完了済み、未解放項目を機械的に管理する |
| learning/LEARNING_TASK_TRACKER.md | 学習者とエージェント | 学習者の進捗、理解度、未完了項目を管理する |
| learning/LEARNING_HANDOFF.md | 学習者とエージェント | 現時点の学習内容、直近の作業、次にやることを記録する |
| tools/check_lesson_structure.sh | エージェント | 必須ファイルと配置を機械的に検査する |
| tools/lesson | 学習者とエージェント | レッスンの順番、通過、復習可能範囲を機械的に制御する |
| tools/learn | 学習者とエージェント | 短いコマンドで学習進捗を記録する |

このシナリオでは、補助ドキュメントの内容も本文内に含めます。  
実際のレッスンでは、同じ内容を責務別ディレクトリへ分け、学習者とエージェントが必要な部分だけを参照できるようにします。

## 機械的な強制力

文章のルールだけでは、エージェントや学習者の操作を完全には止められません。  
そのため、このレッスンでは、レッスン順番、進捗記録、構成確認を専用コマンドで制御します。

| 仕組み | コマンド | 強制できること | 限界 |
| --- | --- | --- | --- |
| レッスン順番制御 | `tools/lesson` | 現在の項目だけ開始、通過できる。未来の未完了項目は開始、通過できない。完了済み項目だけ復習として自由に戻れる | Markdownを直接編集する行為までは止められない |
| 構成チェック | `tools/check_lesson_structure.sh` | 必須ファイルの有無、責務別ディレクトリ配置、ルート直下の誤配置を検出する | 手動編集そのものは止められない |
| 学習記録コマンド | `tools/learn` | 学習進捗の書き込み先を `learning/` 配下に限定し、現在のレッスン項目を一緒に記録する | 開発作業用ファイルへの手動編集は止められない |

`tools/lesson` は `lesson/LESSON_FLOW.tsv` を正しい順番の一覧として読み、`learning/LESSON_STATE.tsv` を現在の状態として更新します。  
状態は `current`、`locked`、`completed` の3つです。

```text
current: 今だけ開始、通過できる項目
locked: まだ進めない未来の項目
completed: 通過済みで、自由に復習できる項目
```

より強い強制力が必要な場合は、Gitのpre-commit hook、CI、ファイル権限を追加します。  
このレッスンでは、非エンジニアが扱いやすい範囲として、まず `tools/lesson`、`tools/learn`、`tools/check_lesson_structure.sh` で運用します。

エージェントは、節目で次を実行して構成を確認します。

```text
tools/check_lesson_structure.sh
```

エージェントは、質問を出す前に次を実行して現在の項目を確認します。

```text
tools/lesson 現在地
```

項目を開始するとき:

```text
tools/lesson 開始 day1.mode
```

項目を通過するとき:

```text
tools/lesson 通過 day1.mode "モードと承認ルールを確認した"
```

完了済み項目を見直すとき:

```text
tools/lesson 復習 day1.mode
```

未完了の未来項目を開始、通過しようとすると、コマンドは失敗し、次に進める項目だけを表示します。  
これにより、レッスンのスキップを機械的に防ぎます。

学習者が任意記録をしたいときは、次のように短いコマンドを使います。

```text
tools/learn 現在地
tools/learn 記録 "今日分かったこと"
tools/learn つまずき "分からなかったこと"
tools/learn 中断 "次回はここから再開したい"
tools/learn 理解 "理解できたこと"
```

## デモモードと実作業モード

このレッスンには、説明だけを確認するデモモードと、実際にファイルを作る実作業モードがあります。

| モード | 目的 | ファイル作成 | 使う場面 |
| --- | --- | --- | --- |
| デモモード | 流れを理解する | 作成しない | 講義、確認、リハーサル |
| 実作業モード | AI駆動開発を実践する | 承認後に作成する | 学習者が実際に手を動かすとき |

エージェントは、最初にどちらのモードで進めるか確認します。  
モードが不明な場合はデモモードとして扱い、実ファイル作成や書き込みはしません。

## 目的

1日3時間、1週間で、小さくても達成感のあるタスク管理表を完成させます。

このレッスンの最大の目的は、ユーザーがAI駆動開発の流れを実際に体験することです。  
一問一答で小さく確認しながら進めますが、必要な場面ではユーザーが実際にプロンプトを入力して実行します。  
ただし、レッスンで使うプロンプトはエージェントが適宜用意し、ユーザーは内容を確認してコピペするだけでよい形にします。

完成イメージ:

```text
タスクを追加できる
タスクの状態を変更できる
優先度を付けられる
期限を入れられる
一覧で確認できる
進捗が見える
ブラウザで開いて使える
```

想定する成果物は、最初はシンプルなWebページです。

```text
HTML
CSS
JavaScript
Markdownメモリー
```

サーバーやデータベースは使いません。  
1週間で完成できる範囲に絞ります。

## エージェント進行ルール

エージェントは、この課題を必ず一問一答形式で進めます。

```text
1. 今やることを短く説明する
2. ユーザーに質問を1つだけする
3. ユーザーの回答を待つ
4. 回答に合わせて次の作業を決める
5. 必要ならコピペ用プロンプト、ファイル案、コード案を提示する
6. ユーザーがプロンプトをコピペして実行する
7. 実行結果を確認する
8. ユーザーが了承したら次へ進む
```

エージェントは、一度に大量の質問をしません。

プロンプト提供ルール:

```text
レッスンで使うプロンプトはエージェントが用意する
ユーザーが一からプロンプトを考えなくてよい形にする
そのままコピペできる文面で提示する
目的、対象ファイル、禁止事項、承認条件を含める
書き込みを伴うプロンプトでは、更新案の提示と開発者承認を必須にする
```

フォルダ構成の提示ルール:

```text
作業の節目で全体のフォルダ構成を見せる
今から触るファイルがどこにあるかを示す
新しいファイルを作る前に、追加後の配置イメージを見せる
同期作業では、関係する文書の位置関係を示す
必要以上に細かくせず、ユーザーが全体像を把握できる範囲に絞る
```

禁止:

```text
勝手に仕様を大きくする
未確認のまま実装方針を決める
ユーザーにプロンプトを一から考えさせる
目的や対象が曖昧なプロンプトを実行させる
開発者の承認なしにファイルへ書き込む
開発者メモリを勝手に書き換える
失敗を記録せずに流す
1週間で終わらない規模に広げる
```

推奨:

```text
短く確認する
選択肢を2つか3つに絞る
非エンジニア向けに説明する
コピペで使えるプロンプトを用意する
全体のフォルダ構成を適宜見せる
毎日、小さく動く成果物を残す
迷ったら小さい方を選ぶ
```

## プロンプト実行体験の進め方

このレッスンでは、説明を聞くだけではなく、ユーザーが実際にプロンプトを入力してAIエージェントを動かします。  
ただし、プロンプト作成そのものを学習の負担にしないため、エージェントが必要なプロンプトを用意します。

ユーザーが行うこと:

```text
エージェントが用意したプロンプトを読む
問題なければコピーする
指定された場所へ貼り付けて実行する
実行結果を見て、了承または修正希望を伝える
```

エージェントが行うこと:

```text
今なぜそのプロンプトを使うのかを短く説明する
そのままコピペできるプロンプトを提示する
実行前に注意点を伝える
実行結果を見て、次の一問を出す
必要なら修正版プロンプトを再提示する
```

プロンプトを使う主な場面:

```text
フォルダ構成を確認するとき
学習進捗を記録するとき
学習内容を引き継ぐとき
要件を整理するとき
仕様案を作るとき
実装計画案を作るとき
TASK_TRACKER.mdとHANDOFF.mdへ記録するとき
3文書と作業履歴を同期するとき
整合性チェックをするとき
ふりかえりをするとき
```

非エンジニア向けには、プロンプトは「AIエージェントへの作業依頼書」と説明します。  
ユーザーは依頼書を一から書かず、エージェントが用意した依頼書を確認して使うことで、AI駆動開発の進め方に集中できます。

## 学習進捗管理

このレッスンでは、開発作業の進捗とは別に、学習者本人の学習進捗を記録します。  
開発作業用の `TASK_TRACKER.md` と `HANDOFF.md` は「何を作ったか」を管理します。  
学習用の `LEARNING_TASK_TRACKER.md` と `LEARNING_HANDOFF.md` は「何を学び、どこまで理解し、次に何を学ぶか」を管理します。

非エンジニア向けには、次のように説明します。

```text
TASK_TRACKER.md: 作るものの進み具合
HANDOFF.md: 作る作業の引き継ぎ
LEARNING_TASK_TRACKER.md: 学習者の理解と進み具合
LEARNING_HANDOFF.md: 学習者が次回どこから再開するか
```

### 2つの記録方法

学習進捗は、次の2つの方法で記録します。

| 記録方法 | 目的 | 実行タイミング |
| --- | --- | --- |
| 節目の自動記録 | レッスンの進行に合わせて漏れなく記録する | Day開始、Step完了、3文書作成、同期完了、Day終了 |
| 任意の短いコマンド | 学習者が気になったタイミングで記録する | 分かった、つまずいた、中断した、再開したいとき |

ここでいう自動記録は、専用コマンドが `learning/` 配下に直接記録するという意味です。  
`tools/lesson` と `tools/learn` を学習者または開発者が明示的に実行した場合、その実行自体を承認済みの機械記録として扱います。  
一方、エージェントがMarkdown本文を直接編集する場合や、開発作業用ファイル、設計文書を更新する場合は、これまでどおり更新案を提示し、開発者の承認を待ちます。

実作業モードでは、レッスン開始時に次のどちらで進めるかを確認します。

```text
節目ごとに確認してから記録する
このレッスン中の学習進捗だけは、節目ごとに自動記録してよい
```

後者を選んだ場合でも、自動記録の対象は `learning/LESSON_STATE.tsv`、`LEARNING_TASK_TRACKER.md`、`LEARNING_HANDOFF.md` だけです。  
開発作業用ファイルや設計文書は、引き続き更新案と承認を必要とします。

### 節目の自動記録

エージェントは、次の節目で学習進捗の記録案を作ります。

```text
レッスン開始時
各Dayの開始時
各Dayの完了時
REQUIREMENTS.md、SPECIFICATION.md、IMPLEMENTATION_PLAN.mdを作成したとき
3文書とTASK_TRACKER.md、HANDOFF.mdを同期したとき
学習者がつまずいたとき
レッスンを中断するとき
レッスンを再開するとき
```

記録する内容:

```text
現在のDayまたはStep
学習者が理解したこと
まだ不安なこと
実行したプロンプト
作成または確認したファイル
次にやること
次回の最初の一問
```

### 任意記録用の短いコマンド

学習者は、必要なタイミングで次の短いコマンドを入力できます。

| コマンド | 目的 | 書き込み先 | 実行時の扱い |
| --- | --- | --- | --- |
| `tools/learn 現在地` | 今どこまで進んだか確認する | 書き込みなし | 直接表示する |
| `tools/learn 記録 "..."` | 今の学習状態を残す | `LEARNING_TASK_TRACKER.md`、`LEARNING_HANDOFF.md` | 明示実行された機械記録として直接追記する |
| `tools/learn 理解 "..."` | 分かったことを残す | `LEARNING_TASK_TRACKER.md`、`LEARNING_HANDOFF.md` | 明示実行された機械記録として直接追記する |
| `tools/learn つまずき "..."` | 分からない点を残す | `LEARNING_TASK_TRACKER.md`、`LEARNING_HANDOFF.md` | 明示実行された機械記録として直接追記する |
| `tools/learn 中断 "..."` | 次回再開できるようにする | `LEARNING_HANDOFF.md` | 明示実行された機械記録として直接追記する |
| `tools/learn 再開` | 前回の続きから始める | 書き込みなし | 直接表示する |

学習者が自然文で「記録して」と依頼した場合は、エージェントが実行するコマンド案を提示します。  
学習者または開発者がそのコマンドを実行した時点で、`learning/` 配下への機械記録として反映されます。

### 学習進捗用ファイルの役割

| ファイル | 目的 | 主な内容 |
| --- | --- | --- |
| LEARNING_TASK_TRACKER.md | 学習者の進捗管理 | Day別の理解度、完了、未完了、つまずき |
| LEARNING_HANDOFF.md | 学習の引き継ぎ | 現在位置、直近で学んだこと、次にやること、次回の一問 |

学習用の2ファイルは、開発作業の成果物ではありません。  
学習者がAI駆動開発の流れを理解しながら進めるための記録です。

### 学習進捗記録用プロンプト

```text

- 学習者または開発者がtools/learnを明示実行した場合だけ、learning/配下へ直接記録されます
```

## プロジェクトディレクトリとリポジトリの配置

このレッスンでは、まず作業全体の入れ物として `$HOME/projects/` を使います。  
`$HOME` は学習者のホームディレクトリを表すため、ユーザーごとに実際のパスは変わります。  
その直下に、教材リポジトリと成果物リポジトリを並列に置く想定で進めます。  
リポジトリ名や主要ファイルの位置を変えたい場合は、`lesson/LESSON_CONFIG.tsv` を変更します。  
このMarkdown自体はシナリオなので、ここでは実際のフォルダ作成はせず、配置方針を説明します。

プロジェクトディレクトリが必要な理由:

```text
レッスン用の資料、メモ、成果物を同じプロジェクト領域で探しやすくする
別の学習課題や仕事のファイルと混ざらないようにする
AIエージェントに「この範囲を見て作業する」と伝えやすくする
```

教材リポジトリと成果物リポジトリを並列に作る理由:

```text
教材の変更履歴と、実際に作るタスク管理表の変更履歴を分けられる
エージェントが今どちらを編集しているか判断しやすくなる
教材側のGit操作と成果物側のGit操作が混ざりにくくなる
あとで移動、共有、削除するときに範囲が分かりやすくなる
```

## 最小ファイル構成

この課題では、リポジトリ内に最初に次のMarkdownファイル構成を用意する想定で進めます。

```text

TASK_TRACKER.md
HANDOFF.md
SESSION_MEMORY.md
FAILURE_MEMORY.md

```

また、実装に入る前に、必要に応じて次の設計用Markdownを作る想定です。

```text
REQUIREMENTS.md
SPECIFICATION.md
IMPLEMENTATION_PLAN.md
```

学習進捗を管理するため、レッスン用には次のMarkdownも使います。

```text
LEARNING_TASK_TRACKER.md
LEARNING_HANDOFF.md
```

これらのMarkdownは、AIエージェントとの作業を安全に進めるための「作業用ノート」です。  
1つのファイルに全部を書くのではなく、目的ごとに分けることで、あとから見返しやすくなります。

| ファイル | 目的 | 便利になること | 実際に使う場面 |
| --- | --- | --- | --- |
| AGENT.md | エージェントが守るルールを置く | 毎回同じ進め方を保ちやすい | 一問一答、確認が必要な操作、禁止事項を確認するとき |
| TASK_TRACKER.md | 開発作業の進捗を管理する | 今日やることや完了したことが見える | 1週間の作業計画、作業中、完了、確認待ちを整理するとき |
| HANDOFF.md | 作業の引き継ぎを書く | 次回、途中から再開しやすい | 今日はどこまで進んだか、次に何をするかを残すとき |
| SESSION_MEMORY.md | その日の会話や判断を短く残す | 直近の決定を忘れにくい | ユーザーの希望や一時的な判断を記録するとき |
| FAILURE_MEMORY.md | 失敗と再発防止策を残す | 同じ失敗を繰り返しにくい | 一問一答を守れなかった、確認せず進めた、などを記録するとき |
| DEVELOPER_MEMORY.md | 開発者本人だけのメモを置く | まだエージェントに反映しない考えを分けられる | 開発者の思いつきや保留中の方針を残すとき。エージェントは勝手に編集しない |
| LEARNING_TASK_TRACKER.md | 学習者の進捗を管理する | 理解済み、未理解、つまずきが見える | レッスンの節目や任意記録コマンドを使ったとき |
| LEARNING_HANDOFF.md | 学習の現在地を引き継ぐ | 次回どこから再開するか分かる | 中断、再開、Day完了時に現在地を残すとき |

このレッスンでは、エージェントは開発者の承認なしにファイルへ書き込みません。  
`AGENT.md` は進行ルールを確認するための基本ファイルですが、内容を変更する場合は開発者へ確認します。  
特に `TASK_TRACKER.md`、`HANDOFF.md`、`SESSION_MEMORY.md`、`FAILURE_MEMORY.md` は、記録先として使う方針を決めても、実際に編集する前に開発者へ確認します。  
`REQUIREMENTS.md`、`SPECIFICATION.md`、`IMPLEMENTATION_PLAN.md` も、案を提示して開発者が承認してから作成・更新します。  
`DEVELOPER_MEMORY.md` は開発者専用のメモなので、エージェントは内容の提案や参照はしても、勝手に編集しません。

| ファイル | 読める | 更新案を出せる | 承認後に書ける | 勝手に書けない |
| --- | --- | --- | --- | --- |
| AGENT.md | はい | はい | はい | はい |
| TASK_TRACKER.md | はい | はい | はい | はい |
| HANDOFF.md | はい | はい | はい | はい |
| SESSION_MEMORY.md | はい | はい | はい | はい |
| FAILURE_MEMORY.md | はい | はい | はい | はい |
| REQUIREMENTS.md | はい | はい | はい | はい |
| SPECIFICATION.md | はい | はい | はい | はい |
| IMPLEMENTATION_PLAN.md | はい | はい | はい | はい |

| LEARNING_TASK_TRACKER.md | はい | はい | はい | はい |

| DEVELOPER_MEMORY.md | 必要時のみ | はい | いいえ | はい |

`勝手に書けない` は、開発者の承認なしに編集しないという意味です。

## 要件・仕様・実装計画を先に作る理由

実装に入る前に、要件、仕様、実装計画を順番に作ります。  
これは、作り始めてから「思っていたものと違う」とならないようにするためです。

非エンジニア向けには、次のように説明します。

```text
要件: 何を作りたいか
仕様: どう動けばよいか
実装計画: どの順番で作るか
```

このレッスンでは、ユーザーが一問一答で要件を決めます。  
要件が決まったら、エージェントが仕様案と実装計画案を作成します。  
ただし、エージェントが作った案はそのまま確定ではなく、開発者が確認してから使います。

この仕組みにする理由:

```text
ユーザーは「何がほしいか」に集中できる
エージェントは要件をもとに、仕様と作業順に整理できる
実装前に認識違いを見つけやすい
1週間で終わる範囲に絞りやすい
```

設計用Markdownの役割:

| ファイル | 目的 | 作成方法 |
| --- | --- | --- |
| REQUIREMENTS.md | ユーザーが何を作りたいかをまとめる | 一問一答でユーザーの回答から作る |
| SPECIFICATION.md | 画面や動きの具体的な決まりをまとめる | 要件をもとにエージェントが案を作る |
| IMPLEMENTATION_PLAN.md | どの順番で作るかをまとめる | 仕様をもとにエージェントが案を作る |

### 初回作成後と変更時の記録順

最初に `REQUIREMENTS.md`、`SPECIFICATION.md`、`IMPLEMENTATION_PLAN.md` の3文書を作成したら、その作成結果を `TASK_TRACKER.md` と `HANDOFF.md` に記録します。  
以降、`TASK_TRACKER.md` と `HANDOFF.md` は「何を、いつ、どこまで進めたか」を残す作業履歴として使います。

初回の流れ:

```text
1. REQUIREMENTS.md を作る
2. SPECIFICATION.md を作る
3. IMPLEMENTATION_PLAN.md を作る
4. TASK_TRACKER.md に作成済みタスクとして記録する
5. HANDOFF.md に次回引き継ぎとして記録する
```

新機能の追加や不要な機能の削除がある場合は、逆の順番で進めます。  
先に `TASK_TRACKER.md` と `HANDOFF.md` で作業を進め、実装や削除が完了してから、最後に3文書へ反映します。

変更時の流れ:

```text
1. TASK_TRACKER.md に変更タスクを追加する
2. HANDOFF.md に現在の作業状況を残す
3. 新機能の実装、または不要な機能の削除を完了する
4. 完了した内容に合わせて REQUIREMENTS.md を更新する
5. 実際の動きに合わせて SPECIFICATION.md を更新する
6. 実際に行った作業に合わせて IMPLEMENTATION_PLAN.md を更新する
```

この順番にする理由:

```text
初回は、何を作るかを先に決めないと作業を始めにくい
変更時は、まだ完了していない内容を設計文書に確定事項として書かないため
タスクトラッカーとハンドオフに作業途中の判断を残せるため
最後に3文書を更新すると、設計文書が最新の完成状態にそろうため
```

便利になること:

```text
今の完成状態と作業途中の記録を分けて見られる
次回のエージェントが途中から再開しやすい
新機能を入れた理由や、機能を削除した理由が残る
古い設計と実際の画面がずれにくい
```

非エンジニア向けには、3文書は「最新の設計図」、`TASK_TRACKER.md` と `HANDOFF.md` は「作業日誌」と説明します。  
最初は設計図を作ってから作業日誌に記録し、変更するときは作業日誌で進めてから、最後に設計図を最新状態へ直します。

### 同期を体験するためのコピペ用プロンプト

このレッスンでは、3文書と作業日誌の同期をユーザーに実際に体験してもらいます。  
目的は、AIエージェントに「何を読ませ、何を更新させ、どこで確認するか」を具体的に頼めるようになることです。

同期で大事なこと:

```text
先に関係する文書を読む
どの文書を更新するかを明確にする
更新前に差分案を出す
開発者の承認を待つ
更新後に整合性を確認する
```

#### 初回同期用プロンプト

3文書を作成したあと、`TASK_TRACKER.md` と `HANDOFF.md` に記録するときに使います。

```text
REQUIREMENTS.md、SPECIFICATION.md、IMPLEMENTATION_PLAN.mdの内容を確認し、
初回作成が完了した作業としてTASK_TRACKER.mdとHANDOFF.mdへ記録する案を作ってください。

目的:
- 3文書を作成済みであることを作業履歴に残す
- 次回のエージェントが現在の状態を理解できるようにする
- 3文書とTASK_TRACKER.md、HANDOFF.mdの内容に矛盾がないか確認する

ルール:
- まず対象ファイルを読んでください
```

## 回答パターン別の分岐ルール

ユーザーの回答によって、エージェントは次のように進めます。  
迷った場合は、作業範囲を小さくし、質問を1つだけに戻します。

| ユーザーの回答 | エージェントの対応 |
| --- | --- |
| はい | 次のStepへ進む |
| 変更したい | 何を変更したいかを1つだけ聞く |
| 減らしたい | 残すものを1つずつ確認する |
| 増やしたい | 追加候補を聞き、1週間で終わるか確認する |
| 分からない | 推奨案を1つ提示し、それでよいか確認する |
| 後で決めたい | 保留として `HANDOFF.md` 向けに記録案を出す |
| 実行結果が違う | 作業を止め、確認チェックと失敗復旧フローへ移る |

分岐時の基本プロンプト:

```text
回答に合わせて進め方を調整します。
まず変更したい点を1つだけ教えてください。
複数ある場合も、今は一番重要なものから確認します。
```

## 完成の定義

この課題では、成果物としての完成と、学習体験としての完了を分けて確認します。

### 成果物の完成条件

タスク管理表そのものは、次を満たしたら完成扱いにします。

```text
タスク管理表の最小機能が決まっている
ブラウザで開いて使える想定になっている
タスクを追加し、一覧で確認できる流れが説明できる
README.mdで使い方を説明できる
今回やらないことが明確になっている
```

### 学習体験の完了条件

AI駆動開発レッスンとしては、次を満たしたら完了扱いにします。

```text
コピペ用プロンプトを使ってAIエージェントを動かす流れを体験している
全体のフォルダ構成を見て、各ファイルの位置と役割を説明できる
LEARNING_TASK_TRACKER.mdとLEARNING_HANDOFF.mdで学習進捗を確認できる
要件、仕様、実装計画の初期案がある
3文書とTASK_TRACKER.md、HANDOFF.mdを同期する流れを体験している
1週間の開発タスクが管理できている
メモリー用Markdownの役割が明確になっている
ユーザーがタスク管理表の使い方を説明できる
次に改善したいことが1つ残っている
```

実装まで進める場合は、最終的に次を目指します。

```text
index.htmlをブラウザで開く
タスクを追加する
一覧に表示される
状態を確認できる
完成したことをREADMEに書ける
```
